# Codity API Documentation

Welcome to the Codity API! This document outlines everything you need to know to interact with our distributed job scheduler programmatically. Whether you're managing queues, dispatching jobs, or building your own worker clients, you're in the right place.

**Base URL:** `http://127.0.0.1:4000`

---

## 🔐 Authentication

Security first! Before you can interact with any of the core endpoints, you'll need to authenticate. We use JSON Web Tokens (JWT) to keep things secure.

### Create an Account
`POST /auth/register`

Need a new user? Send over an email and password. By default, new users get the `member` role, but you can explicitly ask for `admin` if needed.

| Field | Type | Required | Details |
|-------|------|----------|---------|
| `email` | string | Yes | Your login email. |
| `password` | string | Yes | Keep it secret, keep it safe. |
| `role` | string | No | Either `admin` or `member`. Default is `member`. |

**Success Response (201 Created):**
```json
{
  "token": "eyJhbGciOiJIUzI1NiIs...",
  "user": { "id": 1, "email": "hello@codity.local", "role": "admin" }
}
```

### Log In
`POST /auth/login`

Trade your credentials for a fresh JWT. 

| Field | Type | Required |
|-------|------|----------|
| `email` | string | Yes |
| `password` | string | Yes |

**Success Response (200 OK):**
```json
{
  "token": "eyJhbGciOiJIUzI1NiIs...",
  "user": { "id": 1, "email": "admin@scheduler.local", "role": "admin" }
}
```

> **Friendly Reminder:** For all `/api/*` routes below, make sure to include your token in the headers like this: `Authorization: Bearer <token>`.

## 📊 System Overview

### Get the Big Picture
`GET /api/overview`

Want to know what's happening across the entire system right now? This endpoint serves up a bird's-eye view of your queues, recent jobs, active workers, and health metrics—perfect for building dashboards!

**Success Response (200 OK):**
```json
{
  "queues": [ ... ],
  "jobs": [ ... ],
  "workers": [ ... ],
  "counts": [ { "status": "completed", "count": 11 } ],
  "dlqCount": 3

```

## 📋 Managing Jobs

Jobs are the heartbeat of Codity. Here's how you can create and inspect them.

### List All Jobs
`GET /api/jobs`

Fetch a paginated list of jobs. You can filter this down if you're looking for something specific.

| Query Param | Type | Description |
|-------------|------|-------------|
| `status` | string | Looking for `failed` or `completed` jobs? Pass it here. |
| `queue_id` | int | Filter jobs belonging to a specific queue. |
| `page` | int | Which page to grab (defaults to 1). |
| `limit` | int | How many jobs per page (defaults to 50, max 100). |

### Inspect a Specific Job
`GET /api/jobs/:id`

Dive deep into a single job. This doesn't just return the payload; it gives you the entire lifecycle, including every execution attempt and the raw console logs generated by the workers!

### Dispatch a New Job
`POST /api/jobs`

Time to put the workers to task! Drop a payload into a queue. You can run it immediately, or schedule it for the future.

| Field | Type | Description |
|-------|------|-------------|
| `queue_id` | int | **Required.** Where should this job go? |
| `type` | string | `immediate`, `delayed`, `scheduled`, `recurring`, or `batch`. |
| `payload` | object | The actual data your worker needs to do its job. |
| `priority` | int | Higher numbers jump the line. |
| `run_at` | datetime | Use an ISO string to schedule this job in the future. |
| `cron_expr` | string | Pass a standard cron string for recurring tasks. |

## 📦 Queue Configuration

Queues help you organize and throttle your workloads. (Note: These require `admin` privileges).

### View All Queues
`GET /api/queues`
Returns every queue in the system, along with its configuration.

### Create a Queue
`POST /api/queues`
Need a dedicated pipeline for video processing or email sending? Create it here. You just need a `project_id` and a `name`.

### Update a Queue
`PATCH /api/queues/:id`
Adjust the `priority` or `concurrency_limit` of a queue on the fly. 

### Pause or Resume
`PATCH /api/queues/:id/pause`
`PATCH /api/queues/:id/resume`
Hit the brakes on a queue without losing your jobs. Workers will completely ignore paused queues until you resume them.

## 🔄 Retry Policies

Because things fail sometimes, Codity lets you tell queues exactly how to handle retries.

### Configure Queue Backoff
`PATCH /api/retry-policies/:queueId` *(admin)*

| Field | Type | Description |
|-------|------|-------------|
| `strategy` | string | Choose between `fixed`, `linear`, or `exponential` backoff. |
| `max_attempts` | int | How many times should we try before giving up? |
| `base_delay_ms` | int | The delay multiplier (in milliseconds). |

## 💀 The Dead Letter Queue (DLQ)

When jobs fail too many times, they end up here so they don't clog your active queues.

### View Dead Jobs
`GET /api/dlq`
Grab a list of all permanently failed jobs. If you've configured an OpenAI key, you'll even get an AI-generated summary of exactly why the job failed!

### Give a Job a Second Chance
`POST /api/dlq/:jobId/retry` *(admin)*
Did you fix the bug? Hit this endpoint to rescue a job from the graveyard. It resets the attempt count and drops it right back into the active queue.

## 📈 Real-Time Events & WebSockets

Don't want to poll? We've got you covered. Connect a WebSocket client to `http://127.0.0.1:4000` to get real-time pushes.

| Event | Payload | When it happens |
|-------|---------|-----------------|
| `job:updated` | Job Object | Fired the millisecond a job changes state (e.g. from running to completed). |
| `worker:status` | Worker Object | Fired when a worker sends a heartbeat or goes offline. |

---

## 🛑 Rate Limiting & Errors

To keep the system snappy, all **write** endpoints (POST, PATCH, DELETE) are rate-limited to **120 requests per minute** per IP address.

If something goes wrong, we won't leave you guessing. All errors return a clean JSON object indicating what happened alongside standard HTTP status codes:
`{ "error": "Oops, that queue doesn't exist!" }`
